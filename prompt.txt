PHASE 2.2 IMAGE PREPROCESSING PIPELINE

2.2.1 PREPROCESSING OBJECTIVES
  1. Load raw images (1440×1080 JPG/PNG)
  2. Preserve leaf morphology (aspect ratio must be maintained)
  3. Standardize input size (224×224 for backbone compatibility)
  4. Normalize intensity values (ImageNet mean/std normalization)
  5. Convert to PyTorch tensor format (CHW float32)

2.2.2 PREPROCESSING STEPS (IN ORDER)

  Step 1: Load Image
    Input: File path to image
    Function: load_image_rgb()
    Process:
      - Open image file using PIL.Image.open()
      - Convert to RGB color space (handle RGBA, grayscale, etc.)
      - Verify image loaded successfully
    Output: PIL Image object with RGB channels

  Step 2: Resize with Aspect-Ratio Preservation
    Input: PIL RGB image, target size 224×224
    Function: resize_with_padding()
    Process:
      - Calculate scaling factor: min(224/width, 224/height)
      - Resize image to new dimensions: (width*scale, height*scale)
      - Create 224×224 canvas with black padding (0,0,0)
      - Center-paste resized image on canvas
      - Justification: Prevents leaf shape distortion from hard crops
    Output: PIL image exactly 224×224, leaf centered, aspect ratio preserved

  Step 3: Convert PIL Image to NumPy Array
    Input: PIL Image (RGB, 224×224)
    Function: image_to_numpy()
    Process:
      - Convert PIL image to numpy array
      - Shape transformation: (224, 224, 3) → (3, 224, 224) HWC→CHW
      - Normalize pixel values: [0, 255] → [0.0, 1.0] float32
      - Data type: np.float32
    Output: (3, 224, 224) CHW numpy array, float32, range [0.0, 1.0]

  Step 4: Apply ImageNet Normalization
    Input: (3, 224, 224) CHW numpy array in [0.0, 1.0]
    Function: imagenet_normalize()
    Process:
      - Define ImageNet statistics:
        Mean: [0.485, 0.456, 0.406] per channel
        Std: [0.229, 0.224, 0.225] per channel
      - Apply per-channel normalization: (array - mean) / std
      - Formula for each pixel: normalized_value = (original - mean) / std
    Output: (3, 224, 224) CHW numpy array, float32, range approximately [-2.5, 2.5]

  Step 5: Convert to PyTorch Tensor
    Input: (3, 224, 224) CHW numpy array (ImageNet normalized)
    Process:
      - Convert numpy array to torch.FloatTensor
      - Maintain CHW format
      - Verify dtype: torch.float32
    Output: torch.FloatTensor of shape (3, 224, 224)

2.2.3 END-TO-END PREPROCESSING FUNCTION
  Function name: preprocess_image()
  Input: image_path (string or Path object)
  Output: (3, 224, 224) torch tensor or numpy array
  Execution flow:
    image_path → load_image_rgb() → PIL Image
    PIL Image → resize_with_padding() → 224×224 PIL Image
    PIL Image → image_to_numpy() → (3,224,224) numpy, [0,1]
    numpy → imagenet_normalize() → (3,224,224) numpy, normalized
    numpy → return as tensor or keep as array
  Error handling:
    - Catch PIL open() failures (invalid files)
    - Validate image dimensions not zero
    - Verify output shape before returning


PHASE 2.3 BACKBONE ARCHITECTURE SPECIFICATION

2.3.1 PRIMARY BACKBONE: MobileNetV3-Small

  Model details:
    - Architecture: MobileNetV3-Small with depthwise separable convolutions
    - Parameters: 927,008 total parameters
    - Trainable: No (frozen during training)
    - Feature dimension output: 576-dimensional vectors
    - Input size: 224×224×3 (3-channel RGB)
    - Pretrained on ImageNet: Yes (weights initialize from ImageNet)

  Feature extraction process:
    Input image (1, 3, 224, 224) 
      ↓ Convolutional layers (frozen)
      ↓ Depthwise separable convolutions
      ↓ Inverted residual blocks
      ↓ Global average pooling
      ↓ Output: (1, 576) feature vector

  Why MobileNetV3-Small:
    - Lightweight (927K parameters vs 11.7M for ResNet18)
    - Fast inference (0.06 GFLOPS vs 1.81 for ResNet18)
    - High accuracy (good accuracy-efficiency tradeoff)
    - Suitable for edge deployment
    - Sufficient capacity for leaf classification task

2.3.2 ALTERNATIVE BACKBONES (OPTIONAL)

  MobileNetV2:
    - Parameters: 3,504,872
    - Feature dimension: 1280
    - GFLOPS: 0.30
    - Status: Available as alternative

  EfficientNet-B0:
    - Parameters: 5,288,548
    - Feature dimension: 1280
    - GFLOPS: 0.39
    - Status: Available as alternative

  ResNet18:
    - Parameters: 11,689,512
    - Feature dimension: 512
    - GFLOPS: 1.81
    - Status: Available as reference/upper-bound

2.3.3 BACKBONE LOADING PROCESS

  Function: build_feature_extractor()
  Input: backbone_name string
  Process:
    1. Import torchvision.models
    2. Load pretrained model with weights=None (or use pretrained weights if available)
    3. Extract feature layers (remove classification head)
    4. Return feature extraction layers + feature dimension
  Output: Backbone model and feature_dim integer
  Freezing: All parameters set to requires_grad=False


PHASE 2.4 CLASSICAL HEADS IMPLEMENTATION

2.4.1 HEAD ARCHITECTURE REQUIREMENTS

  All heads receive:
    - Input: (batch_size, 576) frozen feature vectors from MobileNetV3-Small
    - Output: (batch_size, 63) logits for 63 classes
    - Training: On frozen features only (no backbone updates)

  Parameter matching strategy:
    - Target parameter count: ~36,000 trainable parameters
    - Goal: Fair comparison across different head types
    - All heads within 12% of baseline (36,351 params)

2.4.2 HEAD TYPE 1: LOGISTIC REGRESSION

  Class: LogisticRegressionHead
  Parameters: 36,351 (576×63 + 63 bias)
  Architecture: Single linear layer
  Implementation:
    - PyTorch linear layer: nn.Linear(576, 63)
    - Training algorithm: sklearn LogisticRegression
    - Solver: 'lbfgs' (efficient for 63 classes)
    - Max iterations: 1000
    - Regularization: L2 (default sklearn)
    - Scaler: StandardScaler (fit on training data)
  Methods required:
    - __init__: Initialize linear layer and scaler
    - forward(): PyTorch forward pass using linear layer
    - fit(features, labels): Fit sklearn model on scaled features
    - predict(features): Return class predictions
    - predict_proba(features): Return class probabilities
    - count_trainable_params(): Return 36,351
  Training process:
    1. Extract training features from backbone (frozen)
    2. Initialize StandardScaler
    3. Fit scaler on training features
    4. Scale training features using fitted scaler
    5. Initialize LogisticRegression with parameters
    6. Fit LogisticRegression on scaled features and labels
  Inference process:
    1. Take new features
    2. Scale using fitted scaler
    3. Call LogisticRegression.predict() or predict_proba()
    4. Return predictions/probabilities

2.4.3 HEAD TYPE 2: SVM with RBF Kernel

  Class: SVMHead with kernel='rbf'
  Parameters: 36,351 (576×63 + 63 bias from linear layer)
  Architecture: Linear layer + sklearn SVC
  Implementation:
    - PyTorch linear layer: nn.Linear(576, 63)
    - Training algorithm: sklearn SVC
    - Kernel: 'rbf' (Radial Basis Function)
    - C parameter: 1.0 (regularization strength)
    - Gamma: 'scale' (default)
    - Probability: True (for predict_proba)
    - Scaler: StandardScaler (required for SVM)
  Methods required:
    - __init__: Initialize linear layer and scaler
    - forward(): PyTorch forward pass using linear layer
    - fit(features, labels): Fit sklearn SVC on scaled features
    - predict(features): Return class predictions
    - predict_proba(features): Return class probabilities
    - count_trainable_params(): Return 36,351
  Training process:
    1. Extract training features from backbone (frozen)
    2. Initialize StandardScaler
    3. Fit scaler on training features
    4. Scale training features using fitted scaler
    5. Initialize SVC with rbf kernel and parameters
    6. Fit SVC on scaled features and labels
  Inference process:
    1. Take new features
    2. Scale using fitted scaler
    3. Call SVC.predict() or predict_proba()
    4. Return predictions/probabilities
  Note: RBF kernel typically provides better accuracy than linear kernel

2.4.4 HEAD TYPE 3: SVM with Linear Kernel

  Class: SVMHead with kernel='linear'
  Parameters: 36,351 (same as RBF variant)
  Architecture: Linear layer + sklearn SVC
  Implementation:
    - PyTorch linear layer: nn.Linear(576, 63)
    - Training algorithm: sklearn SVC
    - Kernel: 'linear' (linear kernel)
    - C parameter: 1.0
    - Gamma: Not used with linear kernel
    - Probability: True
    - Scaler: StandardScaler (still required)
  Purpose: Baseline for comparison with RBF kernel
  Expected performance: Lower than RBF due to limited capacity
  Training/Inference: Identical to RBF variant, just kernel='linear'

2.4.5 HEAD TYPE 4: Dense MLP

  Class: DenseMLP
  Parameters: 41,023 total
    - Layer 1 (576→64): 576×64 + 64 = 36,928 parameters
    - Layer 2 (64→63): 64×63 + 63 = 4,095 parameters
    - Total: 36,928 + 4,095 = 41,023
  Architecture:
    Input (576)
      ↓ Linear(576, 64)
      ↓ ReLU activation
      ↓ Dropout(0.1)
      ↓ Linear(64, 63)
      ↓ Output (63)
  Implementation:
    - PyTorch nn.Sequential
    - Activation: ReLU (Rectified Linear Unit)
    - Regularization: Dropout (0.1 = 10% drop rate)
    - Optimizer: Adam (lr=0.001)
    - Loss function: CrossEntropyLoss
    - Batch size: Flexible, typically 32 or 64
    - Epochs: 50 (for training on feature vectors)
  Methods required:
    - __init__: Define sequential model
    - forward(x): Apply MLP layers to input
    - count_trainable_params(): Return 41,023
  Training process:
    1. Convert features to torch tensor on device (GPU/CPU)
    2. Initialize Adam optimizer
    3. Initialize CrossEntropyLoss
    4. Loop for 50 epochs:
       a. Zero gradients
       b. Forward pass: logits = model(features)
       c. Compute loss: loss = criterion(logits, labels)
       d. Backward pass: loss.backward()
       e. Update weights: optimizer.step()
  Inference process:
    1. Set model to eval mode
    2. Forward pass without gradients: logits = model(features)
    3. Get predictions: predictions = logits.argmax(dim=1)
    4. Return predictions

2.4.6 COMMON INTERFACE FOR ALL HEADS

  Each head class must provide:
    1. fit(features: np.ndarray, labels: np.ndarray)
       Purpose: Train head on frozen features
       Input: (n_samples, 576) array and (n_samples,) label array
    2. predict(features: np.ndarray) → np.ndarray
       Purpose: Get class predictions
       Output: (n_samples,) array of class indices (0-62)
    3. predict_proba(features: np.ndarray) → np.ndarray
       Purpose: Get class probabilities
       Output: (n_samples, 63) array of probabilities [0, 1]
    4. count_trainable_params() → int
       Purpose: Return number of trainable parameters


PHASE 2.5 FEATURE EXTRACTION PROCESS

2.5.1 BATCH FEATURE EXTRACTION

  Function: extract_features()
  Input: List of image file paths
  Output: (n_samples, 576) numpy array of frozen features
  Process:
    1. Load backbone model (MobileNetV3-Small)
    2. Move backbone to device (GPU/CPU)
    3. Set backbone to eval mode (no dropout, batch norm uses running stats)
    4. Create empty features list
    5. For each image path:
       a. Load and preprocess image using preprocess_image()
       b. Convert to batch tensor (1, 3, 224, 224)
       c. Move tensor to device
       d. Forward pass through backbone (no gradients)
       e. Extract feature vector (1, 576)
       f. Append to features list
    6. Convert list to numpy array (n_samples, 576)
    7. Return as float32 array
  Error handling:
    - If image fails to load: append zero vector (576 zeros)
    - Log failed images for debugging
  GPU acceleration:
    - With GPU: ~2-5 minutes for 2000 images
    - With CPU: ~10-15 minutes for 2000 images


PHASE 2.6 TRAINING DATA PREPARATION

2.6.1 DATA LOADING FROM MANIFESTS

  File source: manifests/splits_v1/fold_{N}_{train,val}.csv
  Columns required: file_path, species, condition
  Process for each fold:
    1. Load fold_N_train.csv (training split)
    2. Load fold_N_val.csv (validation split)
    3. For each row in training CSV:
       a. Read file_path relative to data directory
       b. Look up species and condition in ontology
       c. Create label index: condition_idx * 7 + species_idx
       d. Label range: 0-62 (63 total classes)
    4. Store image paths and label indices separately
    5. Create pairs: (image_paths, labels) for each split

2.6.2 LABEL INDEX MAPPING

  Label creation formula:
    label_index = condition_index * num_species + species_index
    where condition_index ∈ [0, 8] and species_index ∈ [0, 6]

  Example mapping:
    - Aloe Vera + Healthy: condition=0, species=0 → label=0
    - Aloe Vera + Disease: condition=1, species=0 → label=7
    - Azadirachta + Healthy: condition=0, species=1 → label=1
    - Last label: Piper Betle + Distorted: condition=8, species=6 → label=62

  Label order depends on species list order from ontology JSON

2.6.3 BATCH ASSEMBLY FOR TRAINING

  Training batch structure:
    - Features: (batch_size, 576) numpy array
    - Labels: (batch_size,) numpy array of integers 0-62
    - Batch size: All training features at once (no mini-batching needed for sklearn)

  Validation batch structure:
    - Same structure as training batch
    - Used for evaluation only, not updates


PHASE 2.7 HEAD TRAINING EXECUTION

2.7.1 TRAINING SEQUENCE FOR SINGLE FOLD

  Step 1: Load split manifest for fold N
    - File: manifests/splits_v1/fold_N_train.csv
    - File: manifests/splits_v1/fold_N_val.csv
    - Extract image paths and labels

  Step 2: Extract frozen features for training set
    - Input: Training image paths (~1600 images)
    - Output: (1600, 576) feature array
    - Time: ~25-30 minutes on GPU, ~60 minutes on CPU
    - Caching: Can optionally cache to disk for reuse

  Step 3: Extract frozen features for validation set
    - Input: Validation image paths (~400 images)
    - Output: (400, 576) feature array
    - Time: ~6-8 minutes on GPU, ~15 minutes on CPU

  Step 4: Train Logistic Regression head
    - Input: Training features (1600, 576) and labels (1600,)
    - Output: Fitted LR model
    - Time: <1 minute (sklearn is very fast)
    - Evaluate on validation set
    - Record metrics: accuracy, F1, precision, recall

  Step 5: Train SVM (RBF) head
    - Input: Same training features and labels
    - Output: Fitted SVM model
    - Time: 1-5 minutes depending on solver
    - Evaluate on validation set
    - Record metrics

  Step 6: Train SVM (Linear) head
    - Input: Same training features and labels
    - Output: Fitted SVM model
    - Time: 1-3 minutes
    - Evaluate on validation set
    - Record metrics

  Step 7: Train Dense MLP head
    - Input: Same training features converted to tensors
    - Output: Trained MLP model
    - Time: 1-2 minutes (50 epochs)
    - Evaluate on validation set
    - Record metrics

  Step 8: Aggregate results
    - Compile metrics for all 4 heads
    - Format: {head_name: {accuracy: X, f1: Y, precision: Z, recall: W}}
    - Save to reports/classical_baseline_fold{N}.json

2.7.2 METRIC COMPUTATION

  For each head on validation set:
    1. Get predictions: y_pred = head.predict(val_features)
    2. Ground truth: y_true = val_labels
    3. Compute metrics:
       - Accuracy: (y_pred == y_true).mean()
       - F1-score: sklearn.metrics.f1_score(y_true, y_pred, average='weighted')
       - Precision: sklearn.metrics.precision_score(y_true, y_pred, average='weighted')
       - Recall: sklearn.metrics.recall_score(y_true, y_pred, average='weighted')
    4. Weighted averaging: Account for class imbalance (important for 63 classes)

================================================================================
ERROR HANDLING AND TROUBLESHOOTING
================================================================================

COMMON ERRORS AND SOLUTIONS:

Error 1: "ModuleNotFoundError: No module named 'torch'"
  Cause: PyTorch not installed or virtual environment not activated
  Solution:
    1. Verify virtual environment is activated: which python (should show .venv path)
    2. Reinstall PyTorch: pip install torch>=2.3.0
    3. Verify: python -c "import torch"

Error 2: "CUDA out of memory"
  Cause: GPU memory insufficient for batch processing
  Solution:
    1. Use CPU only: Set environment variable before running: CUDA_VISIBLE_DEVICES=""
    2. Or reduce batch size in code (not applicable here as we process all at once)
    3. Or use smaller backbone like MobileNetV2

Error 3: "FileNotFoundError: manifests/splits_v1/fold_0_train.csv"
  Cause: Split manifests not created or in wrong directory
  Solution:
    1. Verify current working directory: pwd
    2. Verify file exists: ls manifests/splits_v1/
    3. Create splits if missing (requires Phase 1 completion)

Error 4: "sklearn solver lbfgs did not converge"
  Warning (not fatal): Logistic Regression didn't fully converge
  Solution:
    1. Increase max_iter in LogisticRegression (already set to 1000)
    2. Try different solver: 'saga' or 'newton-cg'
    3. Check data scaling (already implemented with StandardScaler)

Error 5: "PIL.UnidentifiedImageError: cannot identify image file"
  Cause: Corrupted image or unsupported format
  Solution:
    1. Remove corrupted file from dataset
    2. Verify image format is JPG, PNG, or similar
    3. Re-download image if necessary

Error 6: "AssertionError: label out of range (0, 63)"
  Cause: Label index incorrectly calculated or CSV has invalid species/condition
  Solution:
    1. Verify label_ontology.json has exactly 7 species and 9 conditions
    2. Check CSV for species/condition values not in ontology
    3. Verify label calculation formula: condition_idx * 7 + species_idx
